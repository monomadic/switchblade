# Live playback

These notes are the durable record of how the subsystem works **and of the bugs that shaped it** — the war stories are load-bearing, not history. Read the relevant section before changing this area.

## Live playback

The selected clip decodes **once** at quickview resolution (`quickview_max_*`, capped at source res) into the mipmapped hires texture — the tile samples it downscaled, quickview samples it big; one decoder, one timeline, so quickview opens instantly with no handoff/skip. Runs at animation level `normal`+ (quickview skips the settle delay). **Animation levels** (`animation` config / `--animation` CLI, default `normal`): `none` = snap-everything (every spring/fade completes in one frame), `minimal` = UI tweens only, `normal` = + live video. Gates: `level().ui()/live()`; `--animation full` is a legacy alias for normal (the old `full` grid sheet-cycling is removed; the `--no-anim` CLI flag was removed 2026-07-22). The hovered tile gets a separate tile-size lane in a never-evicted atlas slot. All lanes seek-match the thumbnail's frame (`thumb_seek_fraction`, default 10% of duration via cached meta) and die on target move / focus loss. **The start point is one config** (`thumb_seek_fraction`): it drives BOTH the static thumb extraction (`Recipe::seek_fraction`, encoded in the artifact name so a change regenerates — the default omits the `_sNN` suffix, any other value rebuilds) and the live seek (captured once at startup into `seek_fraction`, so the thumb and the first live frame always agree — the no-jolt handoff). Both are **keyframe grabs** at that fraction (`-noaccurate_seek` **+ `-copyts`** for the thumb; keyframe-start with no decode-forward for `SeekablePlayer`), NOT exact seeks: they land on the same keyframe ≤ the fraction, so the handoff still matches frame-for-frame while every thumb skips the per-clip GOP decode-forward (a big win for the gen sweep on sparse-keyframe 4K). **`-copyts` is load-bearing and easy to lose:** input `-ss` rebases timestamps so the landing keyframe comes out NEGATIVE, and the jpeg mux silently drops negative-pts frames — without `-copyts` the file written is the first pts≥0 frame, i.e. the EXACT target (decode-forward and all), up to a GOP ahead of where live opens. That shipped as "video skips backward on hover, variable length, every clip" and survived weeks of pts-based verification because showinfo/null-sink probes DO emit the keyframe — **verify seek behavior by comparing PIXELS of the written artifact (SSIM vs reference frames), never by pts logs.** The `SB_HANDOFF_DUMP=<dir>` env var dumps each lane's first served frames as pts-named PPMs + the cached thumb for exactly this kind of field diagnosis. **`SeekablePlayer` speaks content-relative time** (it adds the stream's `start_time`/edit-list offset when seeking and subtracts it when reporting position): the CLI `-ss` thumb seek is content-relative, so without this a file with a non-zero start_time (phone/camera/QuickTime footage) opened the live stream on a *different* keyframe than the thumb — the handoff jolt the whole no-jolt design exists to prevent (`start_time_offset_is_content_relative`). Changing one side to exact without the other reintroduces the jolt. 0.0 opens at the very beginning. Background ffmpeg/ffprobe jobs run `nice -n 10` — user attention wins CPU.

## One live player (`SeekablePlayer`)

Every live lane — selected stream, warm pool, hover — rides **`SeekablePlayer`** (`sb-media/src/seekable.rs`) — in-process libav via `rsmpeg` (DESIGN.md §15, settled + benchmarked 2026-07): resident demuxer + VT session per stream, so `seek(target, exact)` is a demuxer jump + decoder flush (keyframe ~10–30ms; exact = decode-forward, GOP-bound) instead of the CLI era's ~1s respawn, and cold spawns reach first frame ~2× sooner. It replaced an earlier ffmpeg-CLI-pipe player (`LivePlayer`, now **removed** 2026-07-22 — the in-process path soaked a release; git history has it if the revert is ever needed). Contract: frames due-stamped into a small bounded queue (depth 3 = `LIVE_QUEUE_DEPTH` — decode-ahead absorbs keyframe/cold-start spikes), `take_frame` surfaces a frame only once due, so presentation paces on the **render clock**; late frames **re-anchor** instead of accruing debt (debt = stutter-then-fast-forward). Stamps by **pts delta** from a wall-clock anchor (degrades wrong-meta/VFR to the right wall-clock speed), and the whole player speaks **content-relative time** (start_time offset applied on seek, subtracted on position — see the Live playback bullet). Decode gates are meta-driven: VT on macOS **only for h264/hevc/prores** (VP9/AV1 benchmarked *slower* through VT); **hardware scaling** ([perf review 01](docs/perf-reviews/01-live-video-pipeline.md) — cut 4K60→1440p from ~3.7 to ~0.9 cores) for VT codecs with known 8/10-bit 4:2:0 `pix_fmt` via `scale_vt`+`hwdownload` at target res, dims rounded DOWN to mod-8, ±90° via `transpose_vt` (PSNR-verified) — `SeekablePlayer` reuses the exact `hw_scale_vf` string (shared with the ffmpeg thumbnail chain) as its libavfilter graph, so the chains can't drift; anything else takes the sw chain (which `SeekablePlayer` rotates explicitly — libavfilter doesn't autorotate like the CLI; 90/180/270 covered). Missing `pix_fmt` (old cache) → sw chain now + queued reprobe heals next spawn. The bounded queue is what makes pre-warm free: an undrained player stalls its decoder via backpressure after ~4 frames. Drop MUST wake the condvar (`closed` + notify): a stalled player's reader parks on the full-queue wait where nothing else can reach it — without the wake every dropped warm player leaked a thread pinning ~30MB (live video degraded after minutes of browsing). Warm spawn order is a priority (serialized one at a time): right, right+1, down, left — no 'up' (rare); the second-right slot makes repeated right-taps instant. **Continuity handoff (`hover_resume`):** a selected-lane open (warm promotion or fresh spawn) for a clip whose hover preview is/was just playing continues from the PREVIEW's position — keyframe seek in place on promotion, spawn-at-position otherwise — never the thumb anchor: the warm stream spawns at the anchor and sits parked while the preview plays ahead, so promoting its queued frames jumped playback backward by the whole preview duration (~6s measured on a 4s-GOP 4K clip — the "video skips backward on click" bug; the position is tracked while the lane lives and survives its death for as long as the selection stays on that clip, since the open can trail the click by the settle delay; cleared by shuffle/D-swap like every index-keyed thing). Tests: `hover_click_continues_from_the_preview_position` (sb-app), scenario `benchmarks/scenarios/hover_then_select_handoff.toml`. Regression tests (all `seekable::tests`): `seekable_player_paces_frames`, `unwatched_seekable_player_stalls_then_serves_instantly`, `dropped_seekable_player_releases_its_reader`, `seek_jumps_in_place_without_respawn`, `start_time_offset_is_content_relative`.

## Neighbor decoder pre-warm (the warm pool)

Neighbor decoders pre-warm for the four movement destinations (right, right+1, down, left) in quickview AND in the grid (when live video is on): `warm` holds parked-by-backpressure players, the outgoing stream demotes to warm instead of dying, promotion serves a queued frame the same tick — so single-step selection moves show video instantly. Two ordering rules, both load-bearing: promotion MUST run before the pool is pruned to the new destinations (the selection is never its own neighbor; pruning first killed the warmed decoder and made every advance pay full cold-spawn latency, ~1s on 4K — a cold spawn's floor is probe+VideoToolbox init+first GOP, benchmarked unfixable via flags); and fresh warm spawns wait for settle + the selected stream's first frame, then run one at a time (user attention owns the CPU).

## Focus pause

Losing window focus **parks** the selected stream and the warm pool (undrained → bounded backpressure stalls the decoders within a few frames; stream objects and timelines survive), drops the hover lane, and lets the idle throttle engage; refocus resumes the SAME decoders in place — late frames re-anchor, so playback continues from where it stopped (the old reap-on-defocus made every refocus pay a cold respawn that visibly jumped back to the thumb frame). No spawns of any kind while paused. `pause_unfocused` config (default true), `p` toggles at runtime. Test: `focus_pause_parks_and_resumes_the_same_stream`.

## Skip (`[` / `]`)

An in-place `player.seek(target, exact=false)` on the resident decoder — the stream object, its timeline, and `first_frame` all survive; no respawn, no checkpoint machinery (the old `warm_skip`/`skip_delta` pre-warm dance is deleted). The new offset wraps mod duration; `SelLive::position()` is the decoder's real pts (the seek target while one is in flight — the bar shows where playback is *going*). `hires_shown` (path of the frame in the hires texture) keeps the last frame on screen only through the keyframe jump+flush (~10–30ms) — near-instant, no freeze. **Skips are keyframe seeks** (changed 2026-07-21 from exact): the landing keyframe shows immediately instead of decode-forwarding across the GOP (~500ms on sparse-keyframe 4K — the mpv-style feel the user asked for); the small drift off the exact fraction is invisible for a jump. Same for chapter click / chapter step (`chapter_seek`/`chapter_step` pass `exact=false`). The seekbar scrub stays two-phase: keyframe while dragging + **exact** on release (a precise landing where you let go). Amount = `skip_fraction` tuning, or per-binding `amount` on an internal command. A slim position bar flashes over the video (selected tile or quickview modal) for `SKIP_BAR_HOLD_S` then fades. Test: `skip_seeks_the_selected_stream_in_place` (fixture is `-g 30` so keyframe seeks have a keyframe to land on). The runtime `SeekablePlayer::seek(_, exact)` API still supports exact for the scrub-release path.
