# Grid layout, feel & zoom

These notes are the durable record of how the subsystem works **and of the bugs that shaped it** — the war stories are load-bearing, not history. Read the relevant section before changing this area.

## Grid layouts (`grid_layout`, default "flexible")

"flexible" = justified rows, filmstrip-style — every tile keeps its clip's true aspect (via `chip_aspect`; 16:9 until known) at the row's shared height; rows fill greedily at the nominal height (`tile_width/height` × zoom) then justify by scaling row height so each row spans the window, capped to `row_height_min/max` × nominal (a capped row leaves end slack instead of ballooning; the last row never grows past nominal). "fixed" (alias "square"; "variable"/"mosaic" alias flexible) = the classic uniform crop-fill grid. The geometry is a per-row structure (`FlexGrid`) memoized behind `FlexKey` (viewport width, zoom, clip count, `grid_rev`, grid tuning) — `grid_rev` MUST bump on anything the key can't see: aspect arrivals, shuffle, the D-swap library clear. `Clip.aspect` persists the thumb's aspect past atlas eviction (eviction must never reflow the grid); ALL grid position/hit/row math goes through the mode-agnostic helpers (`tile_rect`, `row_of`, `row_range`, `row_at_y`, `nearest_in_row`) — never index-arithmetic on `lay.cols`, which stays only as the NOMINAL metric (reflow-crossfade trigger, anim-size gate, budget estimates). Vertical selection moves and the warm pool's "down" target land on the horizontally nearest tile in the adjacent row. Tests: `flexible_rows_justify_true_aspect_tiles`, `flexible_vertical_moves_land_on_the_nearest_tile`, `grid_layout_parses_with_aliases`.

## Grid feel & the gripped-ribbon pinch zoom

Per-clip scale + emphasis springs (hover and keyboard ride the same morph: grid crop-fill → true-aspect cover, UV crop derives from live shape); **pinch = gripped ribbon** (`zoom_ribbon`, default on, flexible grid only — see DESIGN.md "Pointer / trackpad"): the library is one strip of chips wrapped at the window width, gripped at a point ON the chip under the fingers, so that chip stays pinned while neighbours expand outward and a chip pushed past an edge is already entering the next row (continuous wrap, no pop). Rows walk outward from the grip row (`ribbon_walk`), each laid as ONE joined strip at its own height — that is what makes "chips are gap-joined, a visiting chip is a full member of its row, rows are one gap apart" structural instead of animated (the earlier per-tile interpolation of absolute rects is what opened black holes and overlaps). Release quantizes to the nearest SAFE packing (`ribbon_quantize` → row start indices, so rows are a partition by construction): each straddler joins whichever row shows more of it unless that squeezes the row past `row_height_min`, full rows re-justify, the strip's first/last rows keep their slack, and rows may rest indented — **never sling a row sideways to left-align it**. The morph (`RibbonSettle`) is a linear interpolation between two gap-joined strips, so every intermediate frame is one too (affine invariant); it carries only on-screen rows. The settled packing (`RibbonPack`) overrides `build_flex` via `flex_grid` and is rebuilt from membership on aspect arrivals (dropped if it stops fitting) — and it is a **fixed point**: a fresh grip on it moves nothing, which a ragged resting row would break. Bugs worth not re-introducing: the grip must clamp the POINT to the chip (clamping the fractions while keeping the raw pointer x jerks the row sideways on frame one); a row that empties must take its gap with it or it leaves a dead band; a departing chip's copy must exit through the WINDOW edge, not merely one gap past its old row, or an indented/short row strands it in view until install pops it away; and **`build_ribbon_flex` treats the starts as a SEED, not a contract** — a thumb arrival changes an aspect, and a row that stops fitting hands its trailing chip down instead of invalidating the packing (dropping it re-justified the whole grid on the next arrival, which is what "chip 0 snaps back to the far left a moment after it settles" was). Keyboard zoom (`set_zoom`) has no grip, so it drops the pack and returns to the justified grid + the wrap below. Tests: `ribbon_pinch_holds_the_gripped_chip_under_the_cursor`, `ribbon_release_settles_on_a_safe_layout`, `ribbon_settled_layout_is_a_fixed_point`, `ribbon_survives_a_long_run_of_pinches`, `ribbon_handoff_does_not_move_the_grid`, `ribbon_packing_survives_a_thumb_arrival` (the demo library never streams thumbs, so only an explicit arrival catches the re-justify snap), `keyboard_zoom_returns_to_the_justified_grid`. Handoff comparisons must be in VIEWPORT space — installing the packing rebases the scroll, so content coords aren't comparable across it. **A gesture must always end** (`ribbon_gesture_always_ends`): a live ribbon draws its mid-flight form (rows at different edge offsets — a "brick" look) and pins `motion`, which on an unfocused window free-runs the frame loop, so only a pinch that MOVES the zoom refreshes the release timer (macOS sends zero-delta magnification around a gesture, and every event is a no-op at the zoom clamps) and losing focus releases immediately. **Never prepend rows/items into a growing Vec** in the walk (`ribbon_release_stays_linear_on_a_big_library`, 12k clips): it is quadratic and froze the main thread at every release — lay them in reverse and flip once. **The LAYOUT is frozen at the grip's zoom for the whole gesture** (`layout()`): the ribbon draws its own rows, so the layout underneath only feeds hit-tests, atlas budgets and warm neighbours — following the mid-gesture zoom repacked the library on every pinch event (always a `FlexKey` miss) and churned everything downstream, dropping and respawning a warm decoder AND the hover lane per frame (VT sessions at frame rate — the real "grinds to a halt", invisible in Tier A because the demo has no decoders). The warm pool is additionally held (`ribboning`, like the D-swap shield) and grid hover is off during a pinch. `step_ribbon` must run BEFORE `step()` captures `lay`, or the frame that installs the packing seeds the wrap table from the pre-install layout and then glides the whole grid to the new one — a visible slide after the gesture already came to rest. Measure any of this with `benchmarks/scenarios/pinch_ribbon.toml` (the `pinch` step and `SB_DEMO_TILES=<n>` exist for exactly this; `/usr/bin/time -l` on the runner gives peak RSS for allocation claims). **Nothing in the gesture path may be O(library) per frame:** the draw table (`ribbon_main`/`ribbon_second`) is a reused buffer cleared only where written, the release walk sheds the middle of rows more than a viewport away (quantization only reads a row's first and last chip), and the morph builds rows only for the band it will actually draw. 20 gestures at 60k clips: p50 0.58ms, peak RSS 39MB, flat.

## Zoom reflow (keyboard zoom + fixed grid)

In the flexible grid a zoom **wraps** tiles that cross a row boundary (`zoom_wrap`, default on): the crossing tile slides off one window edge and the same clip re-enters on the adjacent row from the opposite edge (justified rows span full width, so the window bounds clip the slide for free — no scissor; `build_frame` emits an exit copy + the enter copy, same clip/texture, uv matches since aspect is unchanged), while tiles that merely re-justify glide to their new slot. Driven by a per-clip `reflow: Vec<TileReflow>` table maintained in `step_reflow` (smoothed displayed rect that chases `tile_rect` + an active `WrapEvent`; **dt-accumulated `elapsed`, never wall-clock**, so it stays frame-rate independent). **In wrap mode the LAYOUT zoom snaps straight to `zoom_target` in one frame** (`step`) — one repack, one wrap per crossing tile, prototype-style tween between two settled layouts; the per-clip glide + wrap slides carry ALL the visible zoom motion (the glide sets `motion` — with the zoom snap it IS the zoom animation, or the loop freezes mid-glide). Springing the zoom instead repacked rows at every intermediate value and **restarted** far-down tiles' wraps several times per glide from already-snapped rects — a flickery ghost-fade, then a snap (the second "looks like a fade" bug). **The table is deliberately NOT reset on `grid_rev`** — a real library bumps it on every thumb/aspect arrival, and resetting snapped the whole grid mid-zoom (the first "it just snaps" bug); it only resizes on clip-count change and is cleared explicitly at the two index-remap sites (shuffle, D-swap). A boundary crossing only *wraps* on a frame where the layout zoom actually moved (`zoomed`); a row change while settled (arrival reshapes, window resize) snaps that one tile — and never touches other tiles' in-flight wraps. Wrap `elapsed` advances for **all** entries each frame (not just visible rows) — an off-screen wrap that never expired would pin `motion` and the render loop hot. Only visible rows create wraps / smooth rects (off-screen clips snap on scroll-in). `zoom_wrap_ms` (220) = slide duration, `zoom_wrap_stagger_ms` (0 = all rows lockstep) = per-row ripple delay, `zoom_reflow_smoothing` (0.25) = re-justify glide rate. The old **column-count crossfade** (`transition`, `zoom_fade_ms`) still runs for the **fixed grid** and for shuffle/D-swap rearranges (grid_rev bumps → reflow reseeds, no wrap), and for the flexible grid when `zoom_wrap = false`; the flexible zoom path suppresses it (`!reflow_on` gate on the `lay.cols != last_cols` trigger). Test: `flexible_zoom_wraps_tiles_across_row_boundaries`. Loading dots + cloud badges + background-jobs progress bar are all just extra tiles. (Grid tiles are static thumbs or live video only — sheet cycling is removed; the shader's second-UV crossfade capability (`uv2`/`frame_fade`) remains in tiles.wgsl, currently fed all-zero, reusable for any two-tap blend.) The jobs bar draws **above** the quickview dim so "still working" stays visible in the modal.

## Unloaded tiles awaiting video show the window background

The selected tile's placeholder body and quickview filmstrip chips fill with `background` (not the old lighter greys `0.055`/`0.03`) so a tile waiting on its thumb or live video never flashes bright.
