# Switchblade — agent notes

GPU-rendered video clip picker: "fzf for videos". Pipe paths in on stdin, fly a grid of clips, act on the selected one.

**[PLAN.md](PLAN.md) is the source of truth** for scope, milestones, and every design decision. Read the relevant section before structural changes; don't re-litigate settled decisions (torus deferral, filesystem cache over SQLite, grid sheet-cycling removed / storyboard sheets on-demand-only, etc.) without the user.

## Status
- **Done:** M0–M6 (see PLAN.md §14) plus the internal quickview (M5's "later" form): `Space` or clicking the selection opens a modal playing the clip **at natural resolution** (the same hires stream the selected tile plays — see below), with a **filmstrip** of neighbors along the bottom (selected centered at `strip_selection_scale`, hovered chips scale to `strip_hover_scale` and play their own tile-size lane; both draw above neighbors like the grid; slides on `strip_snap_strength`, chips clickable; `strip_height`/`strip_gap`). In quickview, actions target the foreground: `h`/`l` advance clips, clicking a chip selects it, **clicking the video steps up to fullview** (its stream is already playing — zero handoff, fullview layers over quickview), clicking the frosted background returns to the grid, and Esc / Space close. The grid is backdrop: dimmed (`quickview_dim`) and **frosted** — the grid layer renders offscreen, walks a mip chain, and draws back blurred (`quickview_blur` = downsample levels, 0 disables; `Frame.blur` carries the split across the App boundary). **Neighbor decoders pre-warm** for the four movement destinations (right, right+1, down, left) in quickview AND in the grid (when live video is on): `warm` holds parked-by-backpressure players, the outgoing stream demotes to warm instead of dying, promotion serves a queued frame the same tick — so single-step selection moves show video instantly. Two ordering rules, both load-bearing: promotion MUST run before the pool is pruned to the new destinations (the selection is never its own neighbor; pruning first killed the warmed decoder and made every advance pay full cold-spawn latency, ~1s on 4K — a cold spawn's floor is probe+VideoToolbox init+first GOP, benchmarked unfixable via flags); and fresh warm spawns wait for settle + the selected stream's first frame, then run one at a time (user attention owns the CPU).
- **GPU-residency drops are retryable:** when the atlas is momentarily full, arriving thumbs/sheets are dropped back to `None` (never latched `Failed`) so they reload when visible — the disk cache (unlimited, nothing expires) makes retries cheap.
- **Media pipeline:** ffmpeg/ffprobe workers behind a **strict four-tier queue** — visible thumbs → meta reprobes (one cheap ffprobe rewrites a pre-`pix_fmt` meta.json so that clip's next live spawn can go hardware — the cache never expires, so old entries heal lazily; never probe on the render thread) → the on-demand storyboard sheet (`request_anim_now`, one clip — the user is in quickview/fullview NOW; it originally queued in a bulk-anim tier below gen, and a thumb-recipe change backed gen up for DAYS, silently starving the storyboard — user-attention work never waits behind a sweep) → library-wide background thumb gen (every ingested file queues a disk-cache-only gen job; the progress bar tracks it). **There is no bulk sheet tier anymore** (removed 2026-07-20, PLAN.md Level 2 note): sheets generate ONLY on demand when quickview/fullview opens a clip — clips never opened never pay for one, and the freed workers finish the thumb sweep (and widen the shuffle/random pool) sooner. (Workers use `-hwaccel videotoolbox` for h264/hevc/prores; anim sheets are g² *seeked single-frame extracts* + one tile pass — NEVER a full-clip `fps=` decode, which cost minutes of multi-core churn per long 4K source and survived app quit) → sidecar cache under `~/Library/Caches/switchblade.noindex/v1/objects/<fp>/` (`.noindex` keeps Spotlight's mdworker off the thousands of jpegs; an old un-suffixed cache is renamed across on startup) (artifacts named by recipe with the *ffmpeg* -q:v value; config `thumb_quality` is 1..10, mapped `-q:v = 12 - q` — e.g. `thumb_fit_640x360_q5.jpg`, `anim_3x3_640x360_q5_fit.jpg` + `meta.json`; anim cells preserve the source's TRUE aspect — fit into the thumb/grid cell box, NOT crop-filled to 16:9 — the `_fit` suffix regenerates the old cropped sheets) → decoded to RGBA **via ffmpeg** (not the image crate — keeps thumb and live-video color math identical; the image crate is header-dims only) → runtime-sized GPU atlas (`AtlasCfg`, from `thumb_width/height` + `atlas_width/height` tuning, startup-only; ≤16384/side — device limits are requested to match, don't assume wgpu's 8192 default).
- **Live playback:** the selected clip decodes **once** at quickview resolution (`quickview_max_*`, capped at source res) into the mipmapped hires texture — the tile samples it downscaled, quickview samples it big; one decoder, one timeline, so quickview opens instantly with no handoff/skip. Runs at animation level `normal`+ (quickview skips the settle delay). **Animation levels** (`animation` config / `--animation` CLI, default `normal`): `none` = snap-everything (every spring/fade completes in one frame), `minimal` = UI tweens only, `normal` = + live video. Gates: `level().ui()/live()`; `--no-anim` and `full` are legacy aliases for normal (the old `full` grid sheet-cycling is removed). The hovered tile gets a separate tile-size lane in a never-evicted atlas slot. All lanes seek-match the thumbnail's frame (`thumb_seek_fraction`, default 10% of duration via cached meta) and die on target move / focus loss. **The start point is one config** (`thumb_seek_fraction`): it drives BOTH the static thumb extraction (`Recipe::seek_fraction`, encoded in the artifact name so a change regenerates — the default reuses the legacy suffix-less filename, any other value rebuilds) and the live seek (captured once at startup into `seek_fraction`, so the thumb and the first live frame always agree — the no-jolt handoff). 0.0 opens at the very beginning. Background ffmpeg/ffprobe jobs run `nice -n 10` — user attention wins CPU.
- **Two players, one contract (`SeekablePlayer` / `LivePlayer`):** every live lane — selected stream, warm pool, hover — rides **`SeekablePlayer`** (`sb-media/src/seekable.rs`) — in-process libav via `rsmpeg` (PLAN.md §15, settled + benchmarked 2026-07): resident demuxer + VT session per stream, so `seek(target, exact)` is a demuxer jump + decoder flush (keyframe ~10–30ms; exact = decode-forward, GOP-bound) instead of the CLI era's ~1s respawn, and cold spawns reach first frame ~2× sooner. The CLI-pipe **`LivePlayer`** stays in sb-media (tests + pacebench keep it compiling) as the instant-revert fallback until the in-process path soaks a release, then dies. Both share the same contract: frames due-stamped into a small bounded queue (depth 3 — decode-ahead absorbs keyframe/cold-start spikes), `take_frame` surfaces a frame only once due, so presentation paces on the **render clock**; late frames **re-anchor** instead of accruing debt (debt = stutter-then-fast-forward). `LivePlayer` stamps 1/fps under CFR-forced `-r` output; `SeekablePlayer` stamps by **pts delta** from a wall-clock anchor (same degradation for wrong-meta/VFR: right wall-clock speed). Decode gates are identical, meta-driven: VT on macOS **only for h264/hevc/prores** (VP9/AV1 benchmarked *slower* through VT); **hardware scaling** (PERF.md — cut 4K60→1440p from ~3.7 to ~0.9 cores) for VT codecs with known 8/10-bit 4:2:0 `pix_fmt` via `scale_vt`+`hwdownload` at target res, dims rounded DOWN to mod-8, ±90° via `transpose_vt` (PSNR-verified) — `SeekablePlayer` reuses the exact `hw_scale_vf` string as its libavfilter graph, so the chains can't drift; anything else takes the sw chain (which `SeekablePlayer` rotates explicitly — libavfilter doesn't autorotate like the CLI; 90/180/270 covered). Missing `pix_fmt` (old cache) → sw chain now + queued reprobe heals next spawn. The bounded queue is what makes pre-warm free: an undrained player stalls its decoder via backpressure after ~4 frames. Drop MUST wake the condvar (`closed` + notify): a stalled player's reader parks on the full-queue wait where nothing else can reach it — without the wake every dropped warm player leaked a thread pinning ~30MB (live video degraded after minutes of browsing). Warm spawn order is a priority (serialized one at a time): right, right+1, down, left — no 'up' (rare); the second-right slot makes repeated right-taps instant. Regression tests: `live_player_paces_frames`, `unwatched_player_stalls_then_serves_instantly`, `dropped_player_releases_its_reader` + the same trio for seekable (`seekable::tests`) and `seek_jumps_in_place_without_respawn`.
- **Slot budget & eviction classes:** statics claim budget center-out; anim (storyboard) slots only ever appear from on-demand quickview/fullview installs, a handful at most. Eviction order = out-of-zone anims → out-of-zone statics → in-zone anims; never in-zone statics or live slots.
- **Grid layouts (`grid_layout`, default "flexible"):** "flexible" = justified rows, filmstrip-style — every tile keeps its clip's true aspect (via `chip_aspect`; 16:9 until known) at the row's shared height; rows fill greedily at the nominal height (`tile_width/height` × zoom) then justify by scaling row height so each row spans the window, capped to `row_height_min/max` × nominal (a capped row leaves end slack instead of ballooning; the last row never grows past nominal). "fixed" (alias "square"; "variable"/"mosaic" alias flexible) = the classic uniform crop-fill grid. The geometry is a per-row structure (`FlexGrid`) memoized behind `FlexKey` (viewport width, zoom, clip count, `grid_rev`, grid tuning) — `grid_rev` MUST bump on anything the key can't see: aspect arrivals, shuffle, the D-swap library clear. `Clip.aspect` persists the thumb's aspect past atlas eviction (eviction must never reflow the grid); ALL grid position/hit/row math goes through the mode-agnostic helpers (`tile_rect`, `row_of`, `row_range`, `row_at_y`, `nearest_in_row`) — never index-arithmetic on `lay.cols`, which stays only as the NOMINAL metric (reflow-crossfade trigger, anim-size gate, budget estimates). Vertical selection moves and the warm pool's "down" target land on the horizontally nearest tile in the adjacent row. Tests: `flexible_rows_justify_true_aspect_tiles`, `flexible_vertical_moves_land_on_the_nearest_tile`, `grid_layout_parses_with_aliases`.
- **Grid feel:** per-clip scale + emphasis springs (hover and keyboard ride the same morph: grid crop-fill → true-aspect cover, UV crop derives from live shape); **zoom reflow** — in the flexible grid a zoom **wraps** tiles that cross a row boundary (`zoom_wrap`, default on): the crossing tile slides off one window edge and the same clip re-enters on the adjacent row from the opposite edge (justified rows span full width, so the window bounds clip the slide for free — no scissor; `build_frame` emits an exit copy + the enter copy, same clip/texture, uv matches since aspect is unchanged), while tiles that merely re-justify glide to their new slot. Driven by a per-clip `reflow: Vec<TileReflow>` table maintained in `step_reflow` (smoothed displayed rect that chases `tile_rect` + an active `WrapEvent`; **dt-accumulated `elapsed`, never wall-clock**, so it stays frame-rate independent). **In wrap mode the LAYOUT zoom snaps straight to `zoom_target` in one frame** (`step`) — one repack, one wrap per crossing tile, prototype-style tween between two settled layouts; the per-clip glide + wrap slides carry ALL the visible zoom motion (the glide sets `motion` — with the zoom snap it IS the zoom animation, or the loop freezes mid-glide). Springing the zoom instead repacked rows at every intermediate value and **restarted** far-down tiles' wraps several times per glide from already-snapped rects — a flickery ghost-fade, then a snap (the second "looks like a fade" bug). **The table is deliberately NOT reset on `grid_rev`** — a real library bumps it on every thumb/aspect arrival, and resetting snapped the whole grid mid-zoom (the first "it just snaps" bug); it only resizes on clip-count change and is cleared explicitly at the two index-remap sites (shuffle, D-swap). A boundary crossing only *wraps* on a frame where the layout zoom actually moved (`zoomed`); a row change while settled (arrival reshapes, window resize) snaps that one tile — and never touches other tiles' in-flight wraps. Wrap `elapsed` advances for **all** entries each frame (not just visible rows) — an off-screen wrap that never expired would pin `motion` and the render loop hot. Only visible rows create wraps / smooth rects (off-screen clips snap on scroll-in). `zoom_wrap_ms` (220) = slide duration, `zoom_wrap_stagger_ms` (0 = all rows lockstep) = per-row ripple delay, `zoom_reflow_smoothing` (0.25) = re-justify glide rate. The old **column-count crossfade** (`transition`, `zoom_fade_ms`) still runs for the **fixed grid** and for shuffle/D-swap rearranges (grid_rev bumps → reflow reseeds, no wrap), and for the flexible grid when `zoom_wrap = false`; the flexible zoom path suppresses it (`!reflow_on` gate on the `lay.cols != last_cols` trigger). Test: `flexible_zoom_wraps_tiles_across_row_boundaries`. Loading dots + cloud badges + background-jobs progress bar are all just extra tiles. (Grid tiles are static thumbs or live video only — sheet cycling is removed; the shader's second-UV crossfade capability (`uv2`/`frame_fade`) remains in tiles.wgsl, currently fed all-zero, reusable for any two-tap blend.) The jobs bar draws **above** the quickview dim so "still working" stays visible in the modal.
- **Perf:** idle render throttling — app returns `Frame.animating`; the loop drops to a 100ms tick when false (~2% core), wakes on input. `animating` means *visually* animating only: pending background jobs and an open ingest producer do NOT keep the loop hot — worker threads fire the `sb-window::Waker` (coalescing; armed with an event-loop proxy in `run()`, exposed via `App::waker()`) after each delivery, turning every completion burst into one prompt redraw while a multi-hour gen sweep or a silent open pipe idles at the 10Hz tick. Live video paces the loop by **deadline**, not display rate: `Frame.redraw_at` carries the earliest `next_due()` across live lanes, the idle path sleeps until it, and a frame landing in a dry queue fires the player's notify — so a 30fps clip presents ~30/s on any refresh rate. `RUST_LOG=sb_app=debug` logs a once-a-second redraw-reason line (frames/s + motion/transition/live/timer tallies) — use it to verify any idle/throttling change. **The loop must never rely on the vsync present as its only frame cap:** an occluded window's `render()` early-returns before presenting, so `ControlFlow::Poll` free-ran `frame()` at ~100% core the whole time the app was backgrounded (machine saturated by the time the user came back — looked like a resume freeze). `WindowEvent::Occluded` forces the idle tick regardless of `animating`, and the Poll path floors the redraw interval at `MIN_FRAME` (4ms, under any real refresh rate) as a runaway guard for any other skipped-present path.
- **Focus pause:** losing window focus **parks** the selected stream and the warm pool (undrained → bounded backpressure stalls the decoders within a few frames; stream objects and timelines survive), drops the hover lane, and lets the idle throttle engage; refocus resumes the SAME decoders in place — late frames re-anchor, so playback continues from where it stopped (the old reap-on-defocus made every refocus pay a cold respawn that visibly jumped back to the thumb frame). No spawns of any kind while paused. `pause_unfocused` config (default true), `p` toggles at runtime. Test: `focus_pause_parks_and_resumes_the_same_stream`.
- **Skip (`[`/`]`):** an in-place `player.seek(target, exact=true)` on the resident decoder — the stream object, its timeline, and `first_frame` all survive; no respawn, no checkpoint machinery (the old `warm_skip`/`skip_delta` pre-warm dance is deleted). The new offset wraps mod duration; `SelLive::position()` is the decoder's real pts (the seek target while one is in flight — the bar shows where playback is *going*). `hires_shown` (path of the frame in the hires texture) keeps the last frame on screen through the GOP-bound decode-to-target (~30–600ms) — freeze-then-jump, not reload. A slim position bar flashes over the video (selected tile or quickview modal) for `SKIP_BAR_HOLD_S` then fades. Amount = `skip_fraction` tuning, or per-binding `amount` on an internal command. Skips are **exact** seeks (honest landings on sparse-keyframe sources); the future seekbar scrub uses keyframe mode while dragging + exact on release. Test: `skip_seeks_the_selected_stream_in_place`.
- **Quickview seekbar + filmstrip pointer feel (M8):** pointer motion over the modal video reveals a slim bar along its bottom edge; `seekbar_hide_s` of stillness fades it over `seekbar_fade_ms` (the skip flash shares the same bar — whichever alpha is higher wins). It thickens `seekbar_height`→`seekbar_hover_height` under the pointer; the invisible hit band is taller than the drawn line. Click = seek; hold-drag = scrub, **two-phase like mpv**: keyframe seeks while dragging (the decoder's latest-wins seek cmd coalesces pointer spam; the fill tracks `position()`, which reports the in-flight target, so the bar stays glued to the pointer mid-decode) and one exact seek on release. Hovering the bar shows a **storyboard preview** — the anim sheet cell nearest the hovered timestamp (`seekbar_thumb_width`, 0 disables); quickview requests the selected clip's sheet on demand (`request_quickview_sheet` → `request_anim_now`, the above-gen queue tier, one clip, disk-cached) — the ONLY way sheets are ever generated. Draw and hit-tests share `quickview_video_rect()` so they can't drift. **Filmstrip scroll:** wheel/trackpad over quickview scrubs the strip (`strip_scroll_sensitivity`, negative flips); with `strip_scroll_selects` (default true) the selection commits to the nearest chip (snap spring centers it, grid follows); off = **peek mode** — the strip pans freely while the selected clip keeps playing (a peeked strip HOLDS; it only homes when the selection actually moves, via chip click or keyboard), and either way the backdrop grid never pans under the modal. Filmstrip chips skip the hover settle delay — hovering a chip is deliberate. Tests: `seekbar_click_and_drag_scrubs_the_stream`, `filmstrip_scroll_commits_selection`, `filmstrip_scroll_peeks_without_selecting`.
- **Random / shuffle / auto-skip (`x`/`s`/`t`):** both randomizers are **cached-only** — `Clip.cached` flips true when the clip's gen-sweep job reports done or any artifact delivers (GenDone carries the path), so landing on a clip never queues on-demand ffmpeg work; the pool widens as the sweep runs. `jump_random` picks a uniformly random other *cached* clip (no-op while none are); `shuffle_library` Fisher–Yates the **cached clips to the front** of the library (unswept clips keep their order after them) and **remaps every index-keyed reference through the permutation** (path→index map, atlas slot owners, live/warm/hover lane `clip` fields, the selection — the selected clip keeps playing and lands somewhere new; hover clears; reflow crossfade covers the rearrange; mid-ingest arrivals append after the shuffled block); **auto-skip** (`toggle_auto_skip`, `auto_skip_s` tuning; `toggle_skip_timer`/`skip_timer_s` stay as legacy config aliases) auto-advances selection once the clip has *played* `auto_skip_s` — but ONLY while quickview or fullview is up: back in the grid the countdown suspends, and any suspension (grid, paused, parked, scrubbing, mid-swap, chapter bar) **re-anchors** `auto_skip_since` every tick so re-entering a modal always starts a fresh countdown, never fires instantly on stale watch time. Countdown starts at first live frame (falls back to selection age + a 3s stall grace when a stream never delivers, plain selection age at non-live animation levels) and resets whenever the selection changes; the advance is the same linear "next" the `next` command performs (wrapping at the end). While the countdown exists, a **timer ring** — a stroked white arc over a soft dark scrim, anchored to the VIDEO's top-right corner (`active_video_rect`; window corner while nothing's decoded): by default it FILLS clockwise from 12 toward the next clip (count-up; `auto_skip_countdown = true` flips to a draining countdown), sized by `auto_skip_ring_radius` tuning (`push_countdown_ring`, gated on `auto_skip_progress()`) — shows it; it vanishes with auto-skip off or the modal closed. The arcs are `Tile.pie` (0 = normal tile; else the shader clips fill AND border — so a border-only circle becomes an arc — to a wedge of |pie|: positive ends at 12 and drains clockwise, negative starts at 12 and grows) — the one non-rect thing tiles.wgsl draws, added for this and reusable for any radial meter. Randomness is std-hasher-seeded xorshift — no rand dependency. Tests: `shuffle_remaps_every_index_keyed_reference`, `auto_skip_advances_and_wraps`, `jump_random_lands_only_on_cached_clips`.
- **Chapter bar (`g`, `chapter_mode`/`chapters`) — a fullview add-on:** filmstrip-styled chips slide up from fullview's bottom edge (pressing `g` anywhere enters fullview first; the bar reuses the strip_* tuning, hover/selected scales, snap spring). One cheap worker probe (ffprobe `-show_chapters`, its own tier below thumbs/reprobes, above `anims_now`; deduped by path) answers "real chapters or not" — the chapter list is deliberately NOT in cached `Meta` (old entries would report "no chapters" forever). **Fullview entry pre-warms everything**: the probe fires per visited clip into the session-lived `chapter_probe` cache, and `request_quickview_sheet`'s want-gate includes fullview so the anim sheet is generating/cached before `g` is pressed — the bar usually opens fully populated; probe answers install via `apply_chapter_plan` whether the bar was waiting or opens later. Real chapter starts win; otherwise **synthesized checkpoints at k/n of the duration: none under a minute (the bar slides back down), 4 over a minute (every 25%), 8 over three minutes, 10 over ten** (`checkpoint_count`). Chip images are **the clip's cached anim sheet cells** (the same storyboard the seekbar hover shows) — nearest cell to each chapter's timestamp (cells are the source's true aspect = the chip's aspect, so the center-crop is now essentially identity, kept only to absorb rounding); NOTHING is extracted per chapter (the v1 per-chip rawvideo extraction was slow, stretched, and is gone; no dedicated atlas slots either). Chips share the strip's fixed height at the SELECTED clip's true aspect (`chapter_chip_w`). Wheel/trackpad pans the strip (clamped via `chapter_pos_bounds`: fits-all → pinned centered; panning only, never seeking); the playing chapter wears the selection border + selected scale; clicking a chip exact-seeks the resident stream (skip-bar flash) and slides the bar down; the fullview seekbar's alpha is dampened by the bar's slide (they'd overlap). Esc peels layers: a deliberately-opened bar, then fullview, then quickview. The state drops outright when fullview exits or the selection leaves the bar's clip; auto-skip holds while the bar is up; probe results stay out of the jobs ledger. **Two ways the bar opens** (both via `open_chapter_bar(peek)`): `g` opens it **sticky** (stays until dismissed, `peek_until = None`); a fullview `move_left`/`move_right` chapter step opens it as a **timed peek** (`peek_until = now + chapter_peek_s`, refreshed each step) that slides itself back down when the window elapses — so stepping chapters flashes the bar to show where you landed. A peek does NOT intercept Esc (it's ephemeral; Esc exits fullview and the peek drops with it); re-opening never downgrades a sticky bar to a peek. `chapter_step` runs off the same plan (`chapter_starts`, via the probe cache even with the bar down — see Keys below), so a step with no plan yet is a plain fraction skip and never peeks. Tests: `chapter_probe_reports_starts_and_duration` (sb-media, vs real muxed chapters), `chapter_bar_checkpoints_and_lifecycle`, `chapter_click_seeks_the_stream` (end-to-end: g → probe → chip click seeks the live stream), `fullview_arrows_step_chapters` (also asserts the step peeks the bar), `chapter_peek_auto_closes_but_g_open_stays`, `chapter_mode_binds_to_g`.
- **Modal views freeze the world behind them:** while quickview or fullview is open the grid backdrop is a STATIC snapshot (`frozen_grid`, captured on the first modal frame, invalidated by resize, dropped when every modal closes). The capture NEVER carries the selected tile's hires reference — the modal keeps playing into that texture, so a captured `hires` tile was a live window in a "frozen" backdrop (the selected tile visibly animated behind the blur); the snapshot uses the static thumb instead. **Backdrop styles** (`quickview_backdrop` default "blur", `fullview_backdrop` default "flat", `backdrop_color`): "blur" = the frosted snapshot + `quickview_dim` tint, "flat" = an opaque `backdrop_color` stage with the grid never built at all — either modal wears either (test: `backdrop_styles_gate_the_grid`). Fullview still skips the quickview layer under it. While frozen, nothing behind the modal simulates: the per-clip spring sweep over the whole library is skipped, the camera snaps to its target instead of gliding (state stays right for exit), reflow transitions are dropped, and grid-directed input is ignored (fullview wheel, pinch-zoom in any modal). Grid/strip hover is gated off under every modal. Test: `modal_backdrop_freezes_the_grid`.
- **User-attention playback guards (the "watching a video is utmost priority" rules):** fullview (and the chapter bar) NEVER hover-plays the hidden grid — the pointer still sits "over" an invisible tile behind the black backdrop, and every pointer rest used to cold-spawn a hover decoder for a clip nobody could see, stealing CPU + the VT media engine from the playing video (same gate for filmstrip chips when fullview covers quickview); fullview's prewarms (chapter probe + on-demand anim sheet) wait for the watched stream's FIRST FRAME via `prewarm_ok` (3s `PREWARM_GRACE_S` so a dead/cooldown stream can't starve the storyboard; levels without live video don't wait) — sheet gen is nine niced ffmpeg decodes and racing them against the interactive cold spawn is priority inversion; and `drain_media`'s upload budget drops 64 → `MEDIA_UPLOAD_BUDGET_LIVE` (12) per frame while the selected stream is presenting — a warm-cache burst staged ~80MB of texture writes between two video presents, a visible hitch. Sheet generation additionally waits out any warm decoder that's still filling (`warm_filling`; the chapter PROBE deliberately doesn't) so the worst case never stacks three decode chains at once. Thumb/anim installs in `drain_media` are **idempotent**: coalescing (P1.2) legitimately delivers the same artifact twice (a D swap re-requests while the first job is in flight), and installing without freeing the clip's existing slot left a stale-owner slot whose later eviction cleared the clip's CURRENT thumb. Tests: `fullview_suppresses_hidden_grid_hover`, `fullview_prewarm_waits_for_the_playing_stream`.
- **Strip chips are true-aspect at fixed height** (filmstrip AND chapter bar): every chip keeps `strip_height` and takes the clip's thumb aspect for width (clamped 9:16..2.4; 16:9 until the thumb lands) — portrait clips get narrower chips, never taller ones. Filmstrip positions therefore come from the cumulative `strip_layout` walk (chip floor(pos) anchors at the window center, shifted by the fractional part of one center-to-center step; hit-tests share it via `strip_chip_at`); `strip_geom`'s width is now only the NOMINAL 16:9 width pacing scroll sensitivity/bounds math. Test: `strip_chips_keep_true_aspect_at_fixed_height`.
- **Unloaded tiles awaiting video show the window background:** the selected tile's placeholder body and quickview filmstrip chips fill with `background` (not the old lighter greys `0.055`/`0.03`) so a tile waiting on its thumb or live video never flashes bright.
- **Drag-out (grid tiles + filmstrip chips):** pressing a tile/chip and pulling past `drag_threshold` (tuning, logical px) turns the press into a native macOS drag — `WindowCommand::BeginDrag {path, image}` crosses the App boundary and sb-window's `drag.rs` starts an `NSDraggingSession` whose pasteboard carries the file URL, so dropping into Finder/other apps behaves exactly like dragging out of Finder. The ghost image is the clip's cached thumb jpeg (`MediaService::cached_thumb_path`), falling back to the OS file icon. Two load-bearing rules: the command must be drained **inside the CursorMoved callback** (sb-window calls `apply_commands` there) because the session is seeded from `NSApp.currentEvent`, which must still be the live mouse-drag; and **quickview-open moved from mouse-down to mouse-up** (`Press.open_quickview`) so dragging the selected tile can't open the modal — a matured drag clears the press (the OS owns the gesture; no MouseUp may arrive). Chips select on press as before, then drag the selected clip. Test: `tile_drag_out_matures_and_click_still_opens_quickview`.
- **Siblings swap (`D`, `open_parent`):** rebuilds the library from the selected clip's parent dir (non-recursive `spawn_dir_reader`), streamed like any ingest. `pending_reselect` holds the clip's path across the swap: update_live neither reaps nor spawns by index while it's set (indices are churning; the shielded stream keeps playing — display gates match by **path**, not index), and when the path streams back in it becomes the selection again (cleared on ingest close if the file vanished). ## Build & run
Building needs the **ffmpeg 8.x dev libraries** (brew `ffmpeg` ships them; `rsmpeg` links via pkg-config) on top of the ffmpeg/ffprobe CLIs the pipeline shells out to.
```sh
cargo run -- --demo                # demo grid: 480 fake tiles (no args + TTY stdin = help)
cargo run -- --no-config ~/Clips   # internal defaults; no config searched/watched (tests, triage)
cargo run -- ~/Clips              # CLI paths (files or dirs; dirs walk when recurse=true)
fd -e mp4 -e mov . ~/Clips | cargo run    # or streamed over stdin (args win if both)
RUST_LOG=debug cargo run           # more logging
```
Keys: `hjkl`/arrows are the context-sensitive `move_left`/`move_down`/`move_up`/`move_right` internal commands (fully remappable, defaults in `[keys]`): in the grid every direction moves the selection (horizontal = linear, row-end wraps to the next row; vertical lands on the nearest tile in the adjacent row). **The vertical axis is a view-depth ladder across the modals, with dead-ends at each end:** in quickview, `move_down` dives into fullview (mirror of clicking the quickview video; the strip stays live underneath) and `move_up` does **nothing** (the ceiling — Esc returns to the grid); in fullview, `move_up` steps back out to the filmstrip quickview (`to_quickview` — brings the strip up even when fullview was entered directly via Tab) and `move_down` does **nothing** (the floor). Neither dead-end falls through to a selection move. In fullview, `move_left`/`move_right` step the playing clip between chapters (the bar's plan: real chapters or synthesized checkpoints; forward wraps, back restarts the current chapter when >2s in, and with no plan yet falls back to a fraction skip) and peek the chapter bar (above); in quickview, `move_left`/`move_right` move the selection along the strip. Test: `vertical_keys_step_between_quickview_and_fullview`. The plain linear `next`/`prev` actions remain bindable and always move the selection (auto-skip fires `next`) · `Enter`/`o` open in mpv · `Space` internal quickview · `[`/`]` skip back/fwd through the playing clip (`skip_fraction` of duration, wraps; a binding can carry its own `amount`) · `D` browse the selected clip's parent dir (siblings view; the playing clip keeps playing and re-selects itself) · `g` chapter bar (fullview add-on: chapter/checkpoint chips slide up filmstrip-style; wheel pans, click jumps the video there and lowers the bar) · `x` jump to a random cached clip · `s` shuffle the cached clips to the front of the grid · `t` toggle auto-skip (auto-advance after `auto_skip_s` of quickview/fullview playback; wraps; timer ring on the video, count-up by default) · `c` copy path · `r` reveal in Finder · `p` toggle focus-pause · `f` fullscreen (the `fullscreen` internal action = macOS native/own Space; a separate `fast_fullscreen` action = mpv-style borderless desktop-sized window via winit `set_simple_fullscreen` — bind either or both; each exits whichever mode is active; `--fullscreen`/`--fast-fullscreen` start that way, queued as a `ToggleFullscreen{fast}` command drained in `resumed()` before the first frame) · `-`/`=`/`0` zoom · `q` quit. Every key (movement included) remappable via `[keys]`/`[commands]` in `switchblade.toml` (`type = "launch"` programs with `{path}`/`{dir}`/`{name}` templates, or internal actions — see `commands.rs` for the list). Trackpad pans without changing selection; pinch zooms; click selects; clicking the selection quickviews. In quickview: moving the pointer over the video reveals the seekbar (click to seek, hold-drag to scrub, hover for storyboard thumbs), the wheel/trackpad scrubs the filmstrip, chips hover-play instantly. iCloud placeholder files get a blue tile + cloud badge and are never read.

## Layout
- `src/main.rs` — thin entrypoint.
- `crates/sb-window` — owns the winit loop + wgpu renderer; defines the `App` trait boundary. Instanced rounded-rect tiles in `tiles.wgsl`.
- `crates/sb-app` — implements `App`: grid model, selection, motion, stdin ingest, tuning hot-reload. Headless of OS/GPU types.
- `crates/sb-media` — worker pool (priority queue), ffmpeg/ffprobe invocation, sidecar cache, fingerprinting, `LivePlayer` (CLI pipe, hover lane) + `SeekablePlayer` (in-process libav, selected/warm lanes).

## Hard rules (distilled from PLAN.md)
- **Boundary:** no winit/wgpu types in `sb-app`. `sb-window` owns the loop and calls into the `App` trait; apps see normalized `InputEvent`s and return a plain `Frame`.
- **stdin is sacred:** streaming, newline + NUL delimited, never block the UI on a slow producer. File stats happen on the ingest thread, not the render path. CLI path args take priority over stdin; non-video extensions are skipped, and so are paths with no file on disk (a valid extension alone never claims a chip — except iCloud placeholders, whose file is legitimately absent); dirs walk when `recurse` (default true). Probe meta records display rotation — consumers must swap decode dims for ±90° clips or portrait phone footage stretches.
- **No text stack until M7.** Labels are window-title or bitmap-font debug quality only.
- **Filesystem cache, no SQLite** (until profiling proves otherwise — PLAN.md §8). Cache entries are keyed by `cache_key` config (startup-only): `size_mtime` (default; survives renames/moves) or `path` (abs path + size + mtime, original MVP key). All disk access goes through `entry_dir()` (lazy path→size_mtime adoption on switch); `--cleanup-cache` accepts *either* keying so a config flip never eats a live cache. Never key off the active fingerprint in maintenance — check both via `fingerprint_with`.
- **Feel constants live in `Tuning`** (`crates/sb-app/src/tuning.rs`), hot-reloaded (mtime poll, 250ms) from the first existing of `./switchblade.toml` → `~/.config/switchblade.toml` → `~/.config/switchblade/config.toml` (`config_path()`; `--init` writes the user copy). Never hardcode a motion/feel value elsewhere — add a field. `--no-config` (`Options.no_config`) skips the search and hot-reload entirely — internal defaults only. **Every sb-app test passes `no_config: true`**: the host's `~/.config/switchblade.toml` otherwise leaks into tests (it really happened — a `selection_scale = 1.02` user config broke a scale assertion).
- Motion must stay frame-rate independent: use `tuning::alpha(k, dt)` for smoothing, never raw per-frame lerps.
- Keep dependencies lean; ask before adding one.
