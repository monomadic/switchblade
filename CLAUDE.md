# Switchblade — agent notes

GPU-rendered video clip picker: "fzf for videos". Pipe paths in on stdin, fly a grid of clips, act on the selected one.

**[DESIGN.md](DESIGN.md) is the source of truth for every design decision** (formerly PLAN.md — §N references are stable); **[TASKS.md](TASKS.md) is the source of truth for open work and priority** (epics + tasks, collected from everywhere). Reflective performance reviews live in [docs/perf-reviews/](docs/perf-reviews/). Read the relevant DESIGN.md section before structural changes; don't re-litigate settled decisions (torus deferral, filesystem cache over SQLite, grid sheet-cycling removed / storyboard sheets on-demand-only, etc.) without the user.

## Status

**Shipped:** M0–M6 and M8 (DESIGN.md §14), plus a large body of work beyond the numbered milestones — internal quickview + fullview modals, filmstrip, seekbar/scrub, on-demand storyboard sheets, chapter bar, auto-skip, shuffle/random, native drag-out, siblings swap, the flexible justified grid, gripped-ribbon pinch zoom, and the ingest gatekeeper + sorted ingest. The **attention interaction model** is the only model as of 2026-07-24 (DESIGN.md §15 — adopted, `classic` deleted, don't re-litigate): ONE hires lane follows attention (the hovered tile while mousing, the selection while keyboard-navigating; `hover_delay_ms` / `live_delay_ms` guard the two sides), a single click on ANY tile opens quickview by promoting that already-running stream, and cmd/shift-click multi-selects as border-only `marked` clips that never play. The grid has no tile-size hover lane; `live_hover`/`start_live` survive for the quickview filmstrip's hover-playing chips. **Open work and its priority order live in [TASKS.md](TASKS.md)** — epics M9 → M7 → M10 → M11.

**Invariants that survive every refactor** (each is a shipped bug that came back the moment it was violated — full stories in the linked files):

- **Warm promotion runs BEFORE the pool is pruned** to the new destinations, or every advance pays full cold-spawn latency (~1s on 4K). *(live-playback)*
- **Dropping a player MUST wake the condvar** — a stalled reader parks on the full-queue wait and leaks a thread pinning ~30MB. The bounded queue is also what makes pre-warm free. *(live-playback)*
- **Verify seek behavior by comparing PIXELS** of the written artifact (SSIM vs reference frames), never by pts logs — pts probes report the keyframe the jpeg mux silently dropped. `-copyts` is load-bearing for exactly this. *(live-playback, media-pipeline)*
- **Nothing in the gesture path may be O(library) per frame**, and never prepend into a growing Vec in the ribbon walk — both froze the main thread. `step_ribbon` must run before `step()` captures `lay`. *(grid)*
- **The reflow table is deliberately NOT reset on `grid_rev`** — a real library bumps it on every thumb arrival, and resetting snaps the whole grid mid-zoom. It is cleared only at the two index-remap sites. *(grid)*
- **User attention owns the CPU:** interactive work never queues behind a library-wide sweep, prewarms wait for the watched stream's first frame, and upload budgets shrink while a stream presents or a gesture is in flight. *(render-loop, media-pipeline)*
- **Never index-arithmetic on `lay.cols`** — all grid position/hit/row math goes through the mode-agnostic helpers. *(grid)*
- **The render thread never touches the filesystem.** Cached meta is read on `MetaService` and a spawn that doesn't have it yet defers a frame (`MetaLookup::Pending`) rather than blocking; the drag ghost's `stat` happens once at mouse-down; anything that takes a path argument on the frame path takes a closure, not a resolved value. One `clip_meta` call froze a frame for 901 ms on SMB. *(live-playback, selection-and-actions)*

## Build & run
Building needs the **ffmpeg 8.x dev libraries** (brew `ffmpeg` ships them; `rsmpeg` links via pkg-config) on top of the ffmpeg/ffprobe CLIs the pipeline shells out to.
```sh
cargo run -- --demo                # demo grid: 480 fake tiles (no args + TTY stdin = help)
cargo run -- --no-config ~/Clips   # internal defaults; no config searched/watched (tests, triage)
cargo run -- ~/Clips              # CLI paths (files or dirs; dirs walk when recurse=true)
fd -e mp4 -e mov . ~/Clips | cargo run    # or streamed over stdin (args win if both)
RUST_LOG=debug cargo run           # more logging
```
Keys: `hjkl`/arrows are the context-sensitive `move_left`/`move_down`/`move_up`/`move_right` internal commands (fully remappable, defaults in `[keys]`): in the grid every direction moves the selection (horizontal = linear, row-end wraps to the next row; vertical lands on the nearest tile in the adjacent row). **The vertical axis is a view-depth ladder across the modals:** in quickview, `move_up` dives into fullview (mirror of clicking the quickview video; the strip stays live underneath) and `move_down` drops **back to the grid**; in fullview, `move_down` steps back out to the filmstrip quickview (`to_quickview` — brings the strip up even when fullview was entered directly via Tab) and `move_up` **dismisses the chapter bar** if one is up (swipes it back down off screen), otherwise does **nothing** (the ceiling dead-end, no fall-through to a selection move). **Pinch gestures ride the same ladder** (`InputEvent::Pinch` in a modal steps instead of zooming the hidden grid; deltas accumulate past `MODAL_PINCH_STEP`): pinch-in (fingers apart) dives quickview→fullview; pinch-out (fingers together) backs fullview→strip and strip→grid. In fullview, `move_left`/`move_right` step the playing clip between chapters (the bar's plan: real chapters or synthesized checkpoints; forward wraps, back restarts the current chapter when >2s in, and with no plan yet falls back to a fraction skip) and peek the chapter bar (above); in quickview, `move_left`/`move_right` move the selection along the strip. Tests: `vertical_keys_step_between_quickview_and_fullview`, `fullview_up_dismisses_the_chapter_bar`. The plain linear `next`/`prev` actions remain bindable and always move the selection (auto-skip fires `next`) · `Enter`/`o` open in mpv · `Space` internal quickview · `[`/`]` skip back/fwd through the playing clip (`skip_fraction` of duration, wraps; a binding can carry its own `amount`) · `D` browse the selected clip's parent dir (siblings view; the playing clip keeps playing and re-selects itself) · `g` chapter bar (fullview add-on: chapter/checkpoint chips slide up filmstrip-style; wheel pans, click jumps the video there and lowers the bar) · `x` jump to a random cached clip · `s` shuffle the cached clips to the front of the grid · `t` toggle auto-skip (auto-advance after `auto_skip_s` of quickview/fullview playback; wraps; timer ring on the video, count-up by default) · `c` copy path · `r` reveal in Finder · `p` toggle focus-pause · `f` fullscreen (the `fullscreen` internal action = macOS native/own Space; a separate `fast_fullscreen` action = mpv-style borderless desktop-sized window via winit `set_simple_fullscreen` — bind either or both; each exits whichever mode is active; `--fullscreen`/`--fast-fullscreen` start that way, queued as a `ToggleFullscreen{fast}` command drained in `resumed()` before the first frame). **macOS Tahoe (26) draws a ~1px contour on every non-native-fullscreen window**; `fast_fullscreen` suppresses it via `set_window_shadow(w, false)` (`setHasShadow` is the only lever — `.borderless` styleMask doesn't touch the contour; the shadow is invisible on a desktop-filling window anyway), restored on exit. Native fullscreen escapes the border via its own Space. · `-`/`=`/`0` zoom · `q` quit. Every key (movement included) remappable via `[keys]`/`[commands]` in `switchblade.toml` (`type = "launch"` programs with `{path}`/`{dir}`/`{name}` templates, or internal actions — see `commands.rs` for the list). Trackpad pans without changing selection; pinch zooms; **hovering a tile plays it** (the attention lane, after `hover_delay_ms`) and **a single click on any tile selects it and opens quickview**, promoting that already-running stream; cmd/shift-click marks instead (border-only, never plays; Esc drops the marks). In quickview: moving the pointer over the video reveals the seekbar (click to seek, hold-drag to scrub, hover for storyboard thumbs), the wheel/trackpad scrubs the filmstrip, chips hover-play instantly. iCloud placeholder files get a blue tile + cloud badge and are never read.

## Benchmarking & instrumentation (`sb-bench`)
Repeatable headless perf runs of the REAL app (Tier A: real decoders, real ffmpeg workers, real cache, no GPU/display) through the `App` trait. Full design in [benchmarks/HARNESS.md](benchmarks/HARNESS.md); source in `crates/sb-app/src/bench.rs` + `crates/sb-app/src/bin/sb-bench.rs`. **The runner only MEASURES — it never judges.** A scenario states intent in prose (never parsed); a run emits the full event stream (`events.jsonl`) + a computed `summary.json` (percentiles, counters, condition times); interpretation is retrospective and agentic. Never add pass/fail thresholds to the code.
```sh
cargo build -p sb-app --bin sb-bench                       # NOT `--bin sb-bench` at root
./target/debug/sb-bench run   benchmarks/scenarios/X.toml --out <dir>   # one run, prints summary
./target/debug/sb-bench bench benchmarks/scenarios/X.toml --reps N --label L  # N child procs → reports/
./target/debug/sb-bench compare <bundleA> <bundleB>        # before/after diff table
./target/debug/sb-bench run X.toml --set live_delay_ms=250.0 --set grid_layout=fixed  # knob sweep, no rebuild
```
**Scenarios** (`benchmarks/scenarios/*.toml`): `[setup]` (inputs/fixtures, animation, viewport, `max_wall_s`) + ordered `[[step]]`s (`wait`, `wait_until {cond,n,timeout}`, `key`, `hover`, `click`, `scroll`, `pinch {delta,n}` — a real gesture is a STREAM of small deltas, one per frame, which is the shape that matters for the ribbon zoom; `SB_DEMO_TILES=<n>` scales the fake library so layout/gesture cost can be measured against library size without a big corpus) + `[validity].require` (mechanical validity only — clean exit + these conds met; NEVER a perf verdict). Hermetic: temp `HOME`, `no_config: true`, one child process per rep (cache root is a process-global `OnceLock`). One scenario per run drives `frame()` through the SHARED `sb-window::schedule` policy — never a fixed cadence (that would manufacture idle redraws and inflate `drain_media` throughput).
**Scheduler instruments** (added 2026-07-24, [perf review 03](docs/perf-reviews/03-slow-disk-scheduler.md)): `JobStart`/`JobEnd` events per worker job (queue tier + `hit`/`made`/`failed` outcome + wall time — the hit/made split is load-bearing, since only `made` pays source I/O), `QueueDepths` sampled per tick (`queue_curve`), `worker_utilisation` (the fastest read on "pool working" vs "pool parked on a throttle"), the `pending_bytes` result-backlog canary, `ingest_seen/admitted/rejected/io_us`, per-phase render-thread attribution (`ns_drain_media`/`ns_ingest_install`/`ns_live`), explicit stall-tail counts (frames over 16/50/100/250/1000ms — a p95 summarizes the tail away by construction), and a 1Hz process-tree RSS/thread canary. **`frame_served` fires per served frame, not per lane** — a first-frame-only event cannot see a starved decoder, since the app's tick times stay clean while playback freezes.
**Blocking-I/O instruments** (added 2026-07-24). Three counter families, one per thread that must never block, because on a network volume every one of them inherits the link's fat tail and *none* of them shows up in `tick_ms` percentiles:
- **`render_stall_*`** ([04](docs/perf-reviews/04-network-storage-scheduler.md)) — blocking fs on the RENDER thread, split by op class: `meta` (`clip_meta`) vs `path` (`cached_thumb_path`). **When a frame's `tick_ms` equals `render_stall_max_us` to within a millisecond, that frame WAS one synchronous fs call** — the signature to look for (measured: 1.49 s in 04, 901 ms in 05). Both classes were then fixed (2026-07-24): `meta` is expected to read **0** — the lookup runs on `MetaService`, and `meta_wait_expired` is its cost side (spawns that gave up waiting; also expected 0) — and `path` should read at most one per mouse-down (the drag ghost).
- **`decode_read_*`** ([05](docs/perf-reviews/05-zoom-storm-scheduler.md)) — wall time the live lane's reader thread blocks in `av_read_frame`. **This is the only instrument that can see "the video is frozen":** a reader parked on a slow read emits no frames, no events and costs no tick time, so every other metric reports a perfectly idle app. Read `_over_100ms`/`_over_1s`, never the mean — 0.10 ms mean across 37k reads in the very run holding a 1,011 ms freeze.
- **`queue_wait_*`** ([05](docs/perf-reviews/05-zoom-storm-scheduler.md)) — how long a request SITS before a worker takes it (thumb vs gen). `JobStart`/`JobEnd` measure how long a job *runs*; "the scheduler thrashes" is a claim about waiting. Paired with `thumb_requests` (tier-1 flood size, bounded by the atlas budget) this is what showed the pool *parked* (util 0.36) while visible thumbs waited seconds. Stamps ride deques kept in lockstep with the queues — `queue_wait_stamps_stay_in_lockstep_through_absorption` guards the desync.

**Atlas capacity instruments** ([05 §7](docs/perf-reviews/05-zoom-storm-scheduler.md)): `visible_tiles_max` against the summary's `atlas_slots` is **the first thing to read on any "zoomed out feels broken" report** — if visible exceeds slots, `request_visible_thumbs` (budget `slots()-8`) never even requests the surplus and it falls to the gen sweep, which no queue tuning fixes. **Measure it with `zoom_out_capacity.toml` and `--set` at the config that ships**: the ratio is layout+config only, so demo tiles answer it in 4s with no I/O — and the hermetic default (144 slots) is *not* what the app runs (repo config 777, `~/.config` 1000; ~484 tiles visible at `zoom_min` on a 1600×1000 viewport, so both real configs fit and only the DEFAULT is over capacity). **This is the trap that produced review 05 §7's withdrawn conclusion: a hermetic harness measures `Tuning::default()`, so any claim about a config-derived quantity is a claim about the harness until re-run with `--set`.** `atlas_full_drops` counts finished artifacts discarded for want of a slot (which resets `Thumb::None` and re-arms the request); it reads **0** at every zoom level, i.e. the tempting request→generate→drop→re-request runaway loop does **not** exist, because the tier-1 budget stays below the slot count. Don't re-derive that theory from `alloc_slot` — the counter already settles it.

**Conditions** (`cond_met`, the measurable timeline anchors): `library_count`, `ingest_closed`, `selected_served` (watched stream's first frame), `hover_served` (hover lane's first frame), `grid_settled`, `cache_thumbs`, `jobs_done`/`sweep_drained` (background-sweep progress in the app's ledger), and the anim-sheet brackets `selected_anim_requested` (request_quickview_sheet fired, past the prewarm gates) → `selected_anim` (sheet decoded + atlas-resident = seekbar storyboard / chapter chips can draw). **`frame_served` and `promotion` events carry `pts`** (the served frame's / handoff's content-relative position), so `events.jsonl` exposes playback-position regressions, not just latencies — this is how the hover→select backward jump was measured and how `hover_then_select_handoff.toml` verifies the continuity handoff. `summary.json` records the met-time of **every** `wait_until` cond (not just the validity-required ones), so bracketing conds decompose a latency — e.g. `selected_anim − selected_served` is the fullview→chips delay; splitting it at `selected_anim_requested` separates gate-cascade cost from generation cost. Add a new anchor by extending `cond_met`; add fixtures via `benchmarks/fixtures/generate.sh` (the 4K corpus is genuine 3840×2160 HEVC60, ~5s GOP — a faithful sparse-keyframe proxy).
**Deeper media timing:** per-cell anim-sheet extract times come from `RUST_LOG=sb_media=debug cargo run …` on the REAL app (`animgen …: N cells in …s, per-cell ms=[…]` in `make_anim`) — `sb-bench` has no logger, so use the app for that layer.
**Fidelity limits:** Tier A supports decode/pacing/latency/cache claims only, comparatively. Upload stalls, present-to-present gaps, blur cost, visible jank are Tier B (`--bench` on the real binary) or the live feel eval. On a fast LOCAL SSD the `taskpolicy -b` disk-band half of a change has little to bite on (files warm in page cache) — disk-starvation claims need an external-drive corpus.
**Worked example:** [benchmarks/reports/chapter_sheet_latency.md](benchmarks/reports/chapter_sheet_latency.md) diagnoses "chapter chips take ~a minute after fullview" end to end — added the two anim conds, wrote `chapter_sheet_fullview.toml`, decomposed the delay (gates ~0s, generation ~35s), and attributed it to gen-sweep CPU contention (NOT disk, NOT gates) via the per-cell log. Both levers were then applied (one ffmpeg process for the whole sheet; the sweep throttle engaged at construction) — warm grid ~a minute → **~2.8s**. One edge case remains open (stone-cold library + instant fullview), tracked in TASKS.md.

## Layout
- `src/main.rs` — thin entrypoint.
- `crates/sb-window` — owns the winit loop + wgpu renderer; defines the `App` trait boundary. Instanced rounded-rect tiles in `tiles.wgsl`.
- `crates/sb-app` — implements `App`: grid model, selection, motion, stdin ingest, tuning hot-reload. Headless of OS/GPU types.
- `crates/sb-media` — worker pool (priority queue), ffmpeg/ffprobe invocation, sidecar cache, fingerprinting, `SeekablePlayer` (in-process libav — every live lane: selected, warm, hover).

## Hard rules (distilled from DESIGN.md)
- **Boundary:** no winit/wgpu types in `sb-app`. `sb-window` owns the loop and calls into the `App` trait; apps see normalized `InputEvent`s and return a plain `Frame`.
- **stdin is sacred:** streaming, newline + NUL delimited, never block the UI on a slow producer. File stats happen on the ingest thread, not the render path. CLI path args take priority over stdin; non-video extensions are skipped, and so are paths with no file on disk and files the gatekeeper's header sniff rejects (a valid extension alone never claims a chip — except iCloud placeholders, whose file is legitimately absent); dirs walk when `recurse` (default true). Probe meta records display rotation — consumers must swap decode dims for ±90° clips or portrait phone footage stretches.
- **No text stack until M7.** Labels are window-title or bitmap-font debug quality only.
- **Filesystem cache, no SQLite** (until profiling proves otherwise — DESIGN.md §8). Cache entries are keyed by `cache_key` config (startup-only): `size_mtime` (default; survives renames/moves) or `path` (abs path + size + mtime, original MVP key). All disk access goes through `entry_dir()` (lazy path→size_mtime adoption on switch); `--cleanup-cache` accepts *either* keying so a config flip never eats a live cache. Never key off the active fingerprint in maintenance — check both via `fingerprint_with`.
- **Feel constants live in `Tuning`** (`crates/sb-app/src/tuning.rs`), hot-reloaded (mtime poll, 250ms) from the first existing of `./switchblade.toml` → `~/.config/switchblade.toml` → `~/.config/switchblade/config.toml` (`config_path()`; `--init` writes the user copy). Never hardcode a motion/feel value elsewhere — add a field. `--no-config` (`Options.no_config`) skips the search and hot-reload entirely — internal defaults only. **Every sb-app test passes `no_config: true`**: the host's `~/.config/switchblade.toml` otherwise leaks into tests (it really happened — a `selection_scale = 1.02` user config broke a scale assertion).
- Motion must stay frame-rate independent: use `tuning::alpha(k, dt)` for smoothing, never raw per-frame lerps.
- **Speed and fluidity win over exactness.** This is "fzf for videos" — the feel of flying through clips is the whole product, so a change that makes motion or playback stutter, freeze, or hitch is unacceptable *even when it buys more correct output*. When the two trade off, keep the app fluid and accept the approximate result. Precedent (2026-07-21): seeks (skip/chapter), storyboard cells, and the thumbnail/live-open frame all grab the **nearest keyframe** instead of decode-forwarding to the exact requested timestamp — the exact frame cost a ~500ms decode-forward freeze on sparse-keyframe 4K (mpv doesn't do this because it keyframe-seeks by default), and landing a few frames off is invisible while the freeze was not. Reach for exact only where it's free or where the user is explicitly waiting on precision (e.g. the seekbar scrub's landing on release). Don't trade jank for accuracy without the user.
- Keep dependencies lean; ask before adding one.

## Conventions
- rustfmt defaults, edition 2024.
- `cargo build` from repo root builds everything; a cwd `switchblade.toml` beats the ~/.config ones, so running from repo root during development uses the repo config (which is also the `--init` template, embedded via include_str!).
- `cargo test` (workspace-wide) runs the regression suite cited throughout this doc; every sb-app test constructs `Options` with `no_config: true` so the host's `~/.config/switchblade.toml` can't leak in (see Hard rules).
